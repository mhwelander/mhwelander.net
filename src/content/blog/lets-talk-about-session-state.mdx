## What is session state, again?

If you already know what session state is, skip ahead -- or refresh your memory by reading the MSDN article https://msdn.microsoft.com/en-us/library/ms178581.aspx

This is how I understand session state: HTTP is a stateless protocol. That means that when I requestÂ https://mhwelander.net/Â (request #1) and subsequently requestÂ https://mhwelander.net/blog/Â (request #2), the server treats me as a magical and interesting stranger each time. ASP.NET session stateÂ is a mechanism by which the server remembers who I am by keeping a short-term record of me. It gives me a session ID when I first start browsing, puts it in a cookie (usually), and hey presto -- the server now has a way to identify me as me each time I make a request. Additional data can be added into session state as I make requests -- for example, if I use my name to fill in a form, the application might store that as a session state value so that it can add my name into marketing campaigns: "Martina, get 20% off motherboards!"

![session-state-cookies](https://mhwelander.net/wp-content/uploads/2016/05/session-state-cookies.png?w=782)

When we talk about session state in an xDBÂ context, this is what we are talking about -- the ASP.NET session state mechanism.

## What is 'out-proc' and 'in-proc'?

You can either store session state data in memory (in-proc, or 'in process'), or you can store it somewhere else -- such as in a SQL database (out-proc, or 'out-of-process').

![cookiedata](./images/2016/cookiedata.png)

In a standard ASP.NET application, there are different out-proc session state providers to choose from -- you can see a comparison of the most popular providers here:Â https://blog.devopsguys.com/2013/07/26/best-performing-asp-net-session-state-providers-2013/.

At time of writing, there areÂ twoÂ session state providers available for SitecoreÂ and xDB -- one that uses SQL, and one that uses MongoDB. xDB requires that session state providers support theÂ Session_EndÂ event, which rules out Redis Cache for the version of Sitecore that is currently available (8.1 Update-2). You should choose the provider that uses a technology that you are able to support and optimize.

## What role does session state play in xDB?

Before xDB, there was Sitecore's Digital Marketing Suite -- or DMS (aww, memories). The DMS was a very "chatty" application that made frequent trips to read from and write to the SQL analytics database during the course of a visitor's session. This had performance implications -- particularly for high-traffic, geographically distributed sites where large amounts of data had to travel across huge distances.

![scaling2](./images/2016/scaling2.png)

By contrast, SitecoreÂ accesses the xDB's collection databaseÂ twiceÂ during the course of a visitor's session -- once at the start, to identify the visitor and load data into sessionÂ if the visitor is already known, and once at the end, to flush session data into the collection database. Whilst the session is ongoing, all data -- pages visited, goals triggered, patterns matched -- are held in session.

Relying on session state to hold important data about visitors and their interactions creates the following requirements:

-   Session state should be managedÂ out of processÂ -- this session stateÂ resilient against ASP.NET errors and IIS restarts, and it is aÂ hard requirementÂ in a clustered environment with multiple content Â delivery servers (I will cover this later)
-   Every content delivery cluster should have aÂ dedicated session state database server
-   The session database server should be asÂ physically closeÂ to the CD servers as possible, and on the sameÂ network

The following example shows two content delivery clusters, each with a dedicated session state server:

![scaled](./images/2016/scaled.png)

In this scenario:

-   A visitor hits a cluster (e.g. Europe) -- a request goes out to the collection database toÂ identifyÂ that contact
-   If they are known, data about the contact is loaded into session and a lock is placed on the contact against the current cluster until the end of the session
-   The visitor browses, bouncing between content delivery servers within the cluster (sticky sessions are not required if session state is managed out of process) -- data about the visitor's interaction is written to the session state database
-   After a period of inactivity, the session expires -- analyticsÂ data is written to the collection database and removed from the session state database, and theÂ lock on the contact against the cluster is released

## Private and shared session state

There are two types of session state --Â privateÂ andÂ shared. An easier way to think of this is thatÂ there are two types of session stateÂ *data*Â being stored; data that isÂ private to a particular interactionÂ and data that isÂ shared between interactionsÂ that overlap. Private session state contains information about theÂ interactionÂ -- such as pages visited and goals triggered on that device. Shared session state contains information about theÂ contactÂ -- such as their e-mail address and engagement plan state.

When you install Sitecore locally, private and shared session state are set to 'in-proc'.Â Private session state is configured where you would configure regular old ASP.NET session state -- inÂ web.config. Shared session state is configured inÂ Sitecore.Analytics.Tracking.config.

This distinction (shared vs private) is required to support a single contact withÂ two concurrent sessions on two different devicesÂ -- for example, you might start browsing a website on your laptop and continue on your phoneÂ *whilst the laptop session is still ongoing*(maybe you had to pee during the last few moments of an E-Bay bidding war). What happens if you moved to a different engagement plan stateÂ during the laptop session, or update the e-mail address that is stored in xDB? The laptop session is still ongoing, so those changes have not made it to the collection database yet.

This is where sharedÂ session state comes in, flexing and looking impressive. Contact data needs to beÂ shared across multiple device sessions, and is therefore stored in shared session state. If I moved into a different engagement plan state or changed my name on my laptop, my toiletÂ phone needs to knowÂ about it immediately --Â *before*Â that information hits the collection database.

Why is it important to share certain data between sessions? From a purely technical standpoint, you do not want data from session 1Â to be lost because session 2, which expiresÂ *after*Â session 1, loaded out-of-date information from the collection database before it started. The diagram below (fromÂ http://doc.sitecore.net/) shows the lifetime of shared session:

![docsite](./images/2016/docsite.png)

From a usability point of view, you are offering a seamless experience across multiple devices -- imagine how unimpressed you would be if the personalization rule for a particular engagement plan state was active on your laptop, but your toilet phone acted like it was living in the past.

Read more about private and shared session state on doc.sitecore.net: https://doc.sitecore.net/sitecore_experience_platform/setting_up__maintaining/xdb/session_state/session_state

### A little note about cluster-forwarding

You may wonder what happens if toilet phone is routed to aÂ *different*Â cluster of content delivery servers (let's say you are in between an east coast and west coast cluster). Worry not -- at the start of your laptop session, a lock was placed on your contact data against a particular cluster. When your second session starts, Sitecore checks the xDBÂ and forwards you to the cluster that your pre-existing session is on.

Read more about cluster forwarding on doc.sitecore.net:Â https://doc.sitecore.net/sitecore_experience_platform/setting_up__maintaining/xdb/server_considerations/server_clusters_and_transferring_contact_sessions

## When and why do I have to use out-proc session state management?

If you are using the xDB and you haveÂ more than one content delivery server per cluster, youÂ MUST USE OUT-PROC SESSION STATE. In-proc is categorically not an option, not even if you use sticky sessions. This is all because of the need to support the sharing of session state information for a contact with two concurrent session on different devices. Let's look more closely at why you cannot solve the problem with sticky sessions:

![inproc](./images/2016/inproc.png)

I visit a website from my laptop and log in. My login details happen to be my unique identifier in the xDB, and it loads my contact information from the collection database.Â This website is backed by three content delivery servers -- my session sticks to content delivery server #1 and will stay there for the duration of my session. Session state data is managed in-proc; private and shared session state data is written to memory. So far, so good.

![sadface2](./images/2016/sadface2.png)

I abandon my laptop and switch to my phone -- I log in, am identified a second time, and a new session starts.Â *Or does it*? If my phone session sticks to a different content delivery server, Sitecore has no way of knowing what is happening to my contact data on content delivery server #1 -- because it has no way of accessing the memory (and therefore the shared session data) of another machine. I could have changed my name to Myrtle McMuffin; until that information makes its way into the collection database, my mobile phone has no idea that this has happened.

At time of writing, I do not know what will happen in this scenario (edit:Â [butÂ Tauqir Malik does](https://mhwelander.net/2016/05/19/lets-talk-about-session-state/#comment-3191)!)Â Â -- either the second session simply cannot get a lock on the contact and the session hangs,Â *or*Â you open yourself up to data inconsistencies when session B overwrites session A, and Myrtle McMuffin disappears forever.

![myrtle](./images/2016/myrtle.png)

This is why Â you must use out-proc session state management in an environment with multiple content delivery servers.

Read more about session state and the xDBÂ on doc.sitecore.net:Â https://doc.sitecore.net/sitecore_experience_platform/setting_up__maintaining/xdb/session_state/session_state

## Hey -- can I mix in-proc for private and out-proc for shared?

Technically yes*,Â *since you can configure different providers for shared and private session state,Â but I cannot point to proof that you gain anything at all from doing so. Shared session state must always be configured to use out-proc session state in a multi-content delivery server environment, and will be your limiting factor.
