## Using parameter items

There is another way to set a component's properties, and this method would allow us to translate our Text component's Text property.

Rather than setting the Text value directly in the rendering's properties,Â create an instance of the Text Parameters template. These configuration items go under theÂ PageSettingsÂ child of the page your component is used on:

It does not matter what you call this item, but it helps if it has the same name as the ID of the component it is referring to -- in our case, that'sÂ HeaderTitle. I am going to set the item's Text field toÂ 'Manage Properties':

You can now tell your component to use data from this parameters item by setting it as the component'sÂ DataSource. This is the HeaderTitle component with its DataSource field set to the HeaderTitle parameters item ID:

One of the biggest advantages of using a parameter item as aÂ *datasource*Â is thatÂ your application is now translatable. Using a parameter item is also a nice way of setting defaults, if multiple components on your page could use the same data source item.

For components that are outputting static, application-specific content (like buttons, titles, and introductory text) storing that content in parameter items is acceptable.Â *Business objects*, however, are probably stored as items in the master database -- in our case, a list of houses. ## Data source components

Now that we have a skeleton page with a translatable title, let's turn our attention back to the list of properties (or houses) we are supposed to be outputting. Where is that data coming from?

First of all, we need to set up a template to hold our House data. I am going to go back to the master database to do this, because these items will be used by my public-facing website, too. Let's create a House template and a number of sample items:

![speak-24](./images/2014/speak-24.png)

![speak-25](./images/2014/speak-25.png)

To get this data into my SPEAK application, I first need to create aÂ datasource component. Because I want to be able to search the list of properties eventually, I am going to choose aÂ SearchDataSource:

![speak-26](./images/2014/speak-26.png)

A datasource component does not actually render anything visible in the browser, but itÂ *does*Â add a bit of Javascript:

```xml
<script data-sc-id="HouseDataSource" data-sc-fields="[]" data-sc-showHiddenItems="false" type="text/x-sitecore-searchdatasource" data-sc-require="/-/speak/v1/controls/searchdatasource.js">
```

Notice that it has a bunch ofÂ data attributes, like data-sc-id. These reflect the properties in the component's rendering parameters.


As with the Text component, I can either set rendering parameters directly on the component, or specify a parameter item data source. For consistency's sake, I am going to create a SearchDataSource Parameters item calledÂ HouseDataSourceÂ under PageSettings and set the SearchDataSource component's datasource to the ID of that item:

![speak-27](./images/2014/speak-27.png)

This parameters item has a bunch of exciting fields, but for now I am only interested inÂ DatabaseÂ andÂ RootItemIdÂ right now. As you can imagine, the Database field tells the component which database to look at, and RootItemId tells it which item to start searching from.

Furthermore, if you look at theÂ Network tabÂ in developer tools and refresh the page (I'm using Google Chrome), you can see that a request is being fired off to theÂ Item Web APIÂ as a result of the data source component being on the page:

## Configuring the list control

In order to display the result of the search data source component on the page, we need a second component that knows how to interpret the results and display them.

Add aÂ ListControlÂ to theÂ ApplicationContent.Main placeholderÂ and open its properties. In theÂ Items dropdown, findÂ `{Binding HouseDataSource.Items}`. This is a Javascript property of the search data source component; you are telling your list control that it should get its data from whatever comes back from that data source component.

![speak-31](./images/2014/speak-31.png)

If you save and refresh the page, you will see that the list component now has 3 rows -- but no data is being displayed:

![speak-32](./images/2014/speak-32.png)

We need to configure which fields should be displayed by the list. To do this, create an item based on theÂ ListControl ParametersÂ template under the PageSettings item -- just as we have already done for the Text and SearchDataSource components (ignore the icon; yours will have a normal grey form icon by default):

![speak-33](./images/2014/speak-33.png)

Under that, add items based on theÂ ColumnFieldÂ template. For each one, set theÂ DataFieldÂ (which is the name of the Sitecore item field you want to output) and theÂ HeaderText, which is the title of the table column.

When you are done, set theÂ HouseListControl data sourceÂ to the ID of the HouseListControl item we just created. The list component knows that it should build columns based on the ColumnField items.

## Summary

In part 2, we looked at component parameters, parameter items, data source components, and list components.

-   For simple application-specific content, like titles and button text, you can use the component's 'Text' property -- but if you set it directly in the Properties dialogue, it isn't translatable.
-   All components have aÂ parameters template. You can create an item based on this template under theÂ PageSettingsÂ item of a page, and link it to the component via theÂ DataSourceÂ field.
    -   Remember to leave the property field empty if you want to use the corresponding field from the parameter item
-   Business objects -- in our case, houses -- can come from another database. These are brought into the SPEAK application via aÂ data source componentÂ -- in our case, aÂ SearchDataSourceÂ component, but there are others (like QueryDataSource)
-   Data source components add a bit of Javascript to the page -- it is then up to another component, such as aÂ ListControlÂ to display the data
